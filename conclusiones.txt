3. FRAMEWORK Y PATRÓN UTILIZADO
================================================================================

Framework: SerenityBDD 4.0.14
Patrón: ScreenPlay

Elegí el patrón ScreenPlay porque organiza el código de manera clara:

- TASKS: Acciones de alto nivel que realiza el usuario
  * OpenPage - Navegar a la página de login
  * Login - Autenticarse con credenciales
  * AddProduct - Agregar producto al carrito
  * ViewCart - Visualizar el carrito
  * Checkout - Completar formulario de información
  * FinishPurchase - Finalizar la compra

- USERINTERFACES: Localizadores de elementos de cada página
  * LoginPage, InventoryPage, CartPage, CheckoutPage

- STEPS: Conexión entre Gherkin y Tasks (PurchaseSteps.java)

4.1 LOCALIZADORES DE ELEMENTOS
------------------------------
Al inspeccionar la página con DevTools (F12), encontré que SauceDemo utiliza
el atributo "data-test" para identificar elementos, lo cual facilita la
automatización y hace los tests más estables.

Ejemplos de localizadores que identifiqué:
- Login: id="user-name", id="password", id="login-button"
- Carrito: data-test="shopping-cart-link"
- Productos: id="add-to-cart-sauce-labs-backpack"
- Checkout: id="first-name", id="last-name", id="postal-code"

5. DIFICULTADES ENCONTRADAS Y SOLUCIONES
================================================================================

PROBLEMA 1: Configuración de Chrome
- Solución: Agregué argumentos en serenity.conf como "--remote-allow-origins=*"

PROBLEMA 2: Elementos no encontrados
- Solución: Usé DevTools (F12) para inspeccionar el HTML real y ajusté
  los selectores usando "data-test" e "id".

PROBLEMA 3: Esperas insuficientes
- Solución: Implementé esperas explícitas con WaitUntil usando isClickable()
  e isVisible().

PROBLEMA 4: Dependencias de Serenity
- Solución: Configuré correctamente el scope de las dependencias en build.gradle.
